工具暴露与调用策略（本轮实现 5 种）
responses_native
适用：OpenAI Responses API（优先）
能力：直接把工具/hosted tool 传给 Responses
runtime 只需监测工具事件并回填结果（或 hosted tool 结果直接在响应中体现）
chat_function
适用：OpenAI Chat / 兼容 OpenAI 的大多数第三方
暴露形式：tools: [{ type:"function", function:{...json schema...}}]
循环方式：后端执行 function call，转 tool role 消息回填
chat_custom
适用：OpenAI Chat 原生支持 custom tool 时
暴露形式：tools: [{ type:"custom", custom:{ name, format:{ text|grammar }}}]
用于：
grammar 约束输入
自由文本输入工具调用
后端解析 custom tool call 并执行统一工具抽象
structured_output_tool_call
适用：模型不稳定/不支持 tools，但支持 structured output
暴露形式：响应格式 JSON（例如 {action:"call_tool", tool_name, args}）
后端解析结构化输出后执行工具，再回填继续循环
prompt_protocol_fallback
最差兼容策略（无 tools / 无 structured output）
纯提示词约束输出一个协议块（例如 JSON fenced block）
后端做清洗/解析/执行/重试
明确标注为低可靠策略，默认禁用，按模型配置启用
关键决策
工具“真实执行”永远走统一 ToolRuntime
模型看到的只是某种“暴露策略”
策略由 provider + apiMode + modelCaps + toolPolicy 自动选择，但可手动覆盖

7. WS 重传与 Provider SSE 断流策略（分开处理）
A. 前后端 WS（调试通道）重传策略
你关心的“断了之后补回来”就是这个。

本轮实现增强：

客户端维护 lastEventSeqByRunId
重连后 subscribe_run 带 lastEventSeq
服务端优先走 WS replay_events_batch
若 gap 太大/内存窗口不足，服务端返回 warning：
code = REPLAY_WINDOW_MISS
并提示客户端：
先拉 GET /api/runs/:runId
再拉 GET /api/trace/:runId/events?afterSeq=...
前端自动补齐，不影响后端继续运行
你问的“分页拉取指令”本质就是这一步的回退机制（不是另一种协议，只是让客户端切到 HTTP 分页补拉）。
